## Project identity

FoodOrderV1 is a mobile-first food-bucket and group-order application:

- **Stack**: React 19, Vite 8, TypeScript (7.0.2 primary + 5.9.3 compatibility), Capacitor 8
  (Android built in CI; iOS project committed), Firebase (Auth, Firestore, Functions callables)
  with a fully functional local-device fallback selected at build time
  (`VITE_FORCE_LOCAL_MODE` / empty `VITE_FIREBASE_*`).
- **Testing**: Vitest + Testing Library (jsdom), Playwright (chromium + mobile-chrome, always
  local mode), Firestore rules tests against the emulator.
- **i18n**: hand-rolled `en`/`ar` catalogs with `translate(locale, key)` lookups and full RTL support.
- **State**: React context (session) + service singletons. No server-state query library,
  no global store library. Do not add either.
- **Version**: `package.json` is the place of record; `functions/package.json` must match
  (`npm run quality:release` gates it).

## Architecture summary

Layered module-first architecture, enforced mechanically by the custom ESLint plugin
(`eslint/architecture-plugin/`, 9 rules at `error` severity, tested in
`tests/eslint/architecture-plugin.test.ts`). Dependency direction is one-way:

```text
src/app          composition only: providers, router, shell
   │
   ▼
src/modules/*    features: auth, buckets, group-orders, orders, social,
                 notifications, dashboard, settings
                 + session      (app-wide user/session/toast/locale state)
                 + data-access  (domain model, service contracts, dual
                                 cloud/local persistence gateways, storage-mode selection)
   │
   ▼
src/shared/*     feature-agnostic ui, helpers, i18n core
src/platform/*   environment, browser, device, network, storage adapters
   │
   ▼
src/packages/*   vendor facades: firebase, router, icons, virtuoso,
                 capacitor-core / -haptics / -network / -preferences / -status-bar
   │
   ▼
vendor npm packages
```

Public surfaces are `@/modules/<name>` and `@/packages/<name>` (each module's `index.ts`).
Deep imports across modules are forbidden. `shared` may import `platform`; nothing below
`app` may import `app`; nothing below `modules` may import `modules`. The legacy tree
(`src/components`, `src/pages`, `src/services`, `src/state`, `src/lib`, `src/hooks`,
`src/config`, `src/i18n`, `src/types`) is migration inventory
(see [docs/migration/](docs/migration/README.md)) and is deleted as it empties.

## Non-negotiable rules digest

Full list: [rules/00-non-negotiable-rules.md](rules/00-non-negotiable-rules.md). The short form:

- One-way layer imports; feature code lives in exactly one module.
- Vendor imports only inside the owning `src/packages/*` facade
  (registry: `eslint/package-ownership.config.mjs`).
- Hooks only in `*.hook.ts` / `hooks/`; `*.component.tsx` files call zero hooks;
  containers call project hooks only.
- Browser globals only in `src/platform`; env reads only in `src/platform/environment`.
- Absolute route strings only in `routes/` files; kebab-case + responsibility suffixes;
  `enum` is banned (`as const` objects).
- Never weaken `firestore.rules`, auth, ownership isolation, privacy, a11y, i18n, or tests.
- `.ai/` is generated; never hand-edit.
- `eslint --max-warnings 0`, both typechecks, all tests, build, knip, madge, audits stay green.

**When a rule fails, the code is in the wrong layer. Move or redesign the code. Do not disable the rule.**

## Validation commands

Run targeted checks first, then widen by risk:

```bash
npx vitest run tests/domain/order.test.ts   # targeted example
npm run lint:fix && npm run lint            # CI requires lint:fix to produce zero diff
npm run typecheck                           # TypeScript 7.0.2 (primary)
npm run typecheck:tsc                       # TypeScript 5.9.3 (compatibility) — BOTH must pass
npm run test                                # Vitest
npm run format:check
npm run build
npm run quality:circular                    # madge
npm run quality:dead-code                   # knip
npm run quality:release                     # root/functions version match
npm run test:e2e                            # Playwright, always VITE_FORCE_LOCAL_MODE=true
npm run test:e2e:critical                   # group-order, order-lifecycle, bucket-pricing
npm run test:rules                          # Firestore rules via emulator (rules changes)
npm run functions:validate                  # when functions/ changed
npm run knowledge:build:incremental && npm run knowledge:validate
node scripts/check-agent-docs.mjs           # governance docs integrity
```

CI (`.github/workflows/ci.yml`) additionally runs `npm run security:audit`, Trivy,
coverage, and the Firebase emulator workflow. Every job must be green (`all-gates-green`).

## Definition of done

A change is done only when all of the following hold:

1. Code sits in its owning layer with kebab-case names and responsibility suffixes; the
   architecture plugin reports zero errors without any new `eslint-disable`.
2. Both typechecks pass; `npm run lint:fix` leaves no uncommitted diff; format:check passes.
3. Direct tests exist and pass (unit for pure logic, e2e for screen behavior, emulator for
   rules); no test was skipped, focused, or weakened.
4. `npm run build`, `quality:circular`, `quality:dead-code`, `quality:release` pass.
5. Affected canonical docs updated and `npm run knowledge:build:incremental` run;
   `docs/migration/` status tables updated when migration state changed.
6. Any deviation is a written exception (see below), never a silent one.
7. Commit message is conventional (`commitlint`); hooks were not bypassed.

## Exception policy

Deviations from the architecture are allowed only as documented exceptions:

- Process and template: [docs/exceptions/README.md](docs/exceptions/README.md).
- Active migration exceptions EXC-1 through EXC-5:
  [docs/migration/unresolved-exceptions.md](docs/migration/unresolved-exceptions.md).

Every exception records rule, reason, mitigation, owner, and a removal condition. An
exception may never silently expand its scope. No exception document means no deviation.

## Rule-failure policy

When a rule fails, the code is in the wrong layer. Move or redesign the code. Do not disable the rule.

Concretely: no `eslint-disable` for `architecture/*` rules, no rule severity downgrades,
no file moved out of the linted layout to dodge enforcement, no widening of the
foundational-import list in `eslint/package-ownership.config.mjs`. If you believe a rule
is genuinely wrong for a case, follow [skills/document-exception.md](skills/document-exception.md)
and get the exception written down first.

## Native-platform policy

- Development host is Windows; the Android SDK is available locally
  (`npm run cap:run:android`, `cd android && ./gradlew assembleDebug`). CI builds and
  releases the Android APK on main/tag builds.
- The `ios/` project is committed but **cannot be validated here** — no macOS environment
  exists for this repo (EXC-5). Never claim iOS build, entitlement, or ATS validation.
  Record iOS-affecting changes honestly in
  [docs/migration/native-security-audit.md](docs/migration/native-security-audit.md).
- After web-affecting changes reach native shells, run `npm run cap:sync`
  (`npx cap sync` refreshes `android/` and `ios/`).

## Package-ownership policy

Every npm dependency imported by application source has exactly one owner directory under
`src/packages/` and one public facade `@/packages/<name>`, registered in
`eslint/package-ownership.config.mjs` and enforced by `architecture/no-raw-package-imports`.
Foundational exceptions: `react`, `react/jsx-runtime` (UI runtime) and `react-dom/client`
(bootstrap, `src/main.tsx` only). `@capacitor/app` and `@capacitor/keyboard` hold reserved
registry entries with no owner module until first web-code use (EXC-2). Tests and tooling
config are outside the registry. Details: [rules/08-package-ownership.md](rules/08-package-ownership.md)
and [context/package-ownership.md](context/package-ownership.md).
